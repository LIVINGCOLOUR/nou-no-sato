# 第1部 技術設計（2026-07-07 決定）

## 1. 選定基準

1. **運用が安い** — 目標は月額0円。ユーザーが増えるまで固定費を持たない。
2. **シンプル** — 自前サーバーを持たない。管理するのは「静的フロント」と「Supabaseプロジェクト」の2つだけ。
3. **一般的な構成** — 情報が多く、詰まったときに調べれば出てくる組み合わせにする。

## 2. 構成

```text
[ブラウザ]
   │  静的ファイル（HTML/CSS/JS）
   ▼
[GitHub Pages]  ←─ git push で自動デプロイ（現状のまま）
   │
   │  supabase-js（HTTPS・匿名キー）
   ▼
[Supabase（無料枠）]
   ├─ Postgres …… 全データ
   ├─ Auth ………… メールのマジックリンク（パスワードレス）
   ├─ RLS ………… 公開範囲をDB層で強制（栽培記録は本人のみ 等）
   └─ Storage …… 写真（Phase 2後半。当面はCSSプレースホルダ継続）
```

- **自前APIサーバーは作らない。** ブラウザから supabase-js で直接 Postgres（PostgREST 経由）を読み書きする。「静的サイト＋BaaS」は個人開発でもっとも一般的な構成のひとつ。
- ビルド工程も導入しない（現状の vanilla JS を維持し、supabase-js は CDN の ESM を利用）。

## 3. 費用

| 項目 | プラン | 月額 |
|---|---|---|
| フロント配信 | GitHub Pages | 0円 |
| DB・認証・API | Supabase Free（DB 500MB / 認証 50,000 MAU / Storage 1GB） | 0円 |
| 独自ドメイン（任意） | 取得する場合のみ | 年1,000〜2,000円程度 |

**無料枠の注意点**: Supabase Free はアクセスが約1週間ないとプロジェクトが一時停止する。対策として GitHub Actions の定期ジョブ（無料）で週2回軽いクエリを打って稼働を維持する。Pro（$25/月）への移行は「ユーザーが付いてから」でよい。

## 4. 認証

- **個人**: メールのマジックリンクのみ（パスワード無し）。登録時に必須なのはニックネームだけ。本名・住所はカラム自体を持たない。
- **団体**: 個人アカウントに団体を紐づける（`groups.owner_id`）。登録直後は非公開（status = pending）で、運営承認後に掲載。UIの「新規登録の流れ（申請→運営審査→承認→掲載）」をそのまま実装する。
- **運営（admin）**: `profiles.role = 'admin'`。団体承認・イベント公開・在来種確認の3つの承認キューを処理する。

## 5. データモデル（テーブル一覧）

詳細は `supabase/schema.sql` を正とする。

| テーブル | 内容 | 読める人 | 書ける人 |
|---|---|---|---|
| profiles | ニックネーム・地域・ステージ・関心・ひとこと | 全員 | 本人 |
| groups | 団体プロフィール（status: pending/approved） | approved は全員、pending は所有者と運営 | 所有者（承認は運営） |
| group_updates | 季節の便り（一方向・返信なし） | 全員（approved 団体分） | 団体所有者 |
| events | イベント（status: pending/published） | published は全員 | 承認済み団体の所有者（公開は運営） |
| event_voices | 参加した人の声 | 全員 | 団体所有者・運営 |
| user_event_actions | 気になる／参加予定（kind 列で区別） | 本人（件数は集計ビューで全員） | 本人 |
| follows | 活動を受け取る | 本人 | 本人 |
| invites | イベントに誘った（自分用メモ） | 本人 | 本人 |
| notes | 栽培記録 | **本人のみ** | **本人のみ** |
| seeds | 在来種（出典つき） | 全員 | 運営 |
| seed_contributions | 在来種の情報提供（審査キュー） | 本人と運営 | 認証ユーザーが作成、運営が判定 |

## 6. 公開範囲 → RLS の対応

Phase 1 で決めたプライバシー方針を、アプリのコードではなく **DBの Row Level Security で強制**する（フロントにバグがあっても漏れない）。

- 栽培記録: `notes` は select/insert/update/delete すべて `user_id = auth.uid()` のみ。公開する手段自体が存在しない。
- 地域表示: 市町村程度の粒度しかカラムに入れない（詳細住所カラムを作らない）。
- 連絡先交換: DM・チャットのテーブルを作らない（作らないものは増やさない、が最大の安全策）。
- 気になる・参加予定の**件数**: 個人の行は本人しか読めないが、件数だけは集計ビュー（`event_counts`）で全員に公開する。

# 第2部 確定済みの製品方針（2026-06 実機確認より）

## 用語（Phase 1 で反映済み）

- 「畑ノート」→「栽培記録」に統一。栽培記録は個人ユーザーの機能で基本非公開。団体機能とは別。

## 仲間ページの位置づけ

- 仲間ページは「個人同士が直接連絡を取り合う場」ではなく、「地域にどんな団体・活動者がいるかを知る」ための場。
- 主な役割は、その団体・活動者の**イベントへアクセスする入口**。
- 住所・詳細な活動場所・直接連絡先は原則出さない。
- ファーストコンタクトは個人間連絡ではなく**イベント参加経由**にする。

## 公式リンク

- 団体の公式ホームページ・Instagram・その他SNSは**団体自身が管理ページで登録**する（第三者による勝手な登録は不可）。
- なりすまし・不適切リンクを防ぐ運用ルールが必要（運営審査と合わせて確認）。

## イベントと団体の双方向導線

- イベント⇔団体の双方向リンクを維持する。イベントが**安全なファーストコンタクトの場**。

## アカウント種別

| アカウント | できること | 発行 |
|---|---|---|
| 個人 | 栽培記録の作成 / イベント閲覧・参加予定 / 農法を学ぶ / 在来種を見る | 一般登録 |
| 団体 | 団体プロフィール管理 / 公式URL・SNS登録 / イベント作成・編集 | 運営審査制 |
| 運営 | 全体管理・承認 | 運営のみ |

- イベント作成は**承認済み団体のみ**。

## 学ぶページの参考コンテンツ方針

- 農法詳細の外部リンクは**運営確認済みのリンクのみ**掲載。単なるリンク集にしない。
- 外部リンクは補助。アプリ本体の価値は「地域・イベント・記録・学び」に置く。
